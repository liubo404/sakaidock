Technical Approach to "Everything is Content"     Edit

1  ABSTRACT


The persistence design for k2 is centered around JCR as the storage mechanism and JPA as the searching mechanism. This deviates from traditional development strategies in that entities are not persisted and retrieved from tables in a database but content nodes are retrieved from a hierarchy within a JCR repository and indexed using carefully designed views that are accessed via JPA. While this may seem like a basic premise, the implications of properly storing and retrieving data become more important as the hierarchy of the data becomes more complex. This hierarchy can contain some aspects of the data without having to explicitly define them. For example, by owning a parent node, a user will also own the children nodes below it. More interesting scenarios include changing the ownership of something like a site by simply moving it to a different parent.  Node ownership is not the only means of access control as each node can specify and inherit an access control list.


Thinking of a Sakai deployment as a large storage of content rather than a model of entities implies that everything in the system is the results of type of authoring effort. To create a site, a user decides what to call the site and the roles that are required to access it. This is simply authoring the details of a sites meta content. As information is added to the site, assignments, resources, quizzes, students, etc, these all become nodes of content that are related to the site. Taking a step back, this approach also shows that sites are just the content of a Sakai deployment in that a deployment's content grows as sites are created and as content is added to those sites. This way of looking at all data as just hierarchical content also provides for an elegant way of handling access and security to that content. Each node of content has an associated Access Control List (ACL) and can inherit from parent nodes just like a file system.


2  IMPLEMENTATION DETAILS


This approach of all data as content has been discussed at a high level but I think the more interesting problem is understanding the technical requirements of implementing this.  A client, in this example a REST call, creates a new site by passing the required information down to the SiteService which in turn calls to JCR to store the data. The details of the site are kept in a json file that is streamed into a node of the repository. Since a site has an owner, the site is stored in the user's environment space within the repository (eg. /userenv/user1/sites/sakai-site-1/sitedef.json). An interesting side effect of this is that JCR exposes the repository over WebDAV which allows the site to be referenced and updated using URLs just as updating any other file in a remote file system. The provided service isn't required for updating a site.

The next step in the process is to create views for looking up the data stored in JCR. Creating relational indexes as views into the content not only adds flexibility but boosts performance over the standard content searching built into JCR. Indexes are created through an asynchronous event observer registered with JCR. The observer creates indexes in a database table using JPA and there should be only 1 table used for searching (flat view). Any pieces of the content that a user is to search by should be included in the index along with a reference to the URL of the content associated to the indexed record. All columns of the index are used for searching and have database indices applied to each allowing for optimal searching. For those familiar with OLAP, these data views can be likened to data cubes. Any bit of content is not limited to a single view and can be referenced in number of views as the point of the view is to provide a means to finding a content end.
