K2 Classloader     Edit

This page describes how the classloaders operate in the Agnostic POC.


Agnostic Classloaders


Ideally, the framework which contains the kernel will include the concepts of shared and resolvable classes, exported packages and locally isolated classes. Arguably, OSGi does provide these concepts, but most frameworks do not. To remain agnostic we have implemented some simple classloaders that support these concepts so the framework will work in any container.


The Classloaders

URLClassloader

URLClassloaders are a standard classloader in the JVM, that is constructed with a list of URLs pointing to classpath elements, that may be either jars or classes directories. We use this class as the basis for all our classloaders.

ComponentClassloader

The component classloader extends the URLClassloader but has a custom search sequence for resolving class. First it tried to resolve classes using classloaders registered against containing packages, failing that it tries local resolution and failing that shared. This means that exported packages take precidence avoiding the ClassCast Exception we have been plagued with in 2.x.

SharedClassloader

The Shared Classloader extends the ComponentClassloader, and so resolves exported classes first. In addition this classloader allows the addition of Shared classpath dependencies at runtime.

WebApp Classloader

The WebApp classloader is used by webapps, but behaves identically to the component classloader.

Container Classloader

The container classloader specifies the group of classloaders provided by the container. In tomacat5 this includes the common, tomcat shared, and server classloaders. The tomcat shared classloader should not be confused with the SharedClassloader above.

 

Classloader Construction

 

 

When a component starts up, it needs a classloader. Component Classloader service is used to take the Component Specification and generate a new classloader to host the component. It creates the classloader using all the local class path elements. It takes the shared classloader and registeres it as the parent of the new Component Classloader. 

In addition to creating the new ComponentClassloader, the Component Classloader Service adds any shared classpath elements to the Shared Classloader. Having done that it registers component classloader against exported packages as specified in the Component Specification.


Classloader Loading Order


 

Once constructed, classes are resolved with a specific order. A request to load  a class comes from one of the beans in the component. The Component Classloader (A)  first tries to locate a classloader that has exported the package stub, if the package stub is bound to a classloader and the class can be resolved, that class is used. Failing that, the class is resolved locally in the component classloader. Failing that the shared classloader is checked.

 

 

Where the class is from an exported package in Classloader B, the PackageRegistryService is consulted and classloader B is found. The class is then resolved from classloader B.


 

If no exported package matches the requested package, the classloader will be null. Then classloader b tires locally. If the class is resolved this class is used.

 

Finally if the class is not resolved due to export or due to local resolution, then the shared classloader or its parents are consulted.
