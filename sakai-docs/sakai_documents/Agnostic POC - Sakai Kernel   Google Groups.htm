Agnostic POC    

This page describes the work being done on an agnostic POC.


Agnostic Kernel POC


The Agnostic POC (agnostic in svn) builds on the OSGi POC (osgiloader in svn), by using the same launch mechanism with MBeans. The structure is similar to the structure described in the Component Manager page, that was written while the work on the OSGi POC was in progress, however there are no external framework dependencies inside the core kernel in an attempt to simplify the kernel and make it easier to work with.


The key concepts/requirements in the Agnostic POC are as follows:

1. The Kernel should not bind to any container technology.

2. The Kernel should be structured in such a way as to make it possible to work in any container technology (WebApp, EBJ, MBeans, GBeans etc)

3. The Kernel should have a mechanism of defining components and the dependencies of a component.

4. The Kernel should support a DAG of dependencies, and be capable of starting a set of components in the correct order dependent on that DAG.

5. The Kernel should have a mechanism of service registration.

6. The Kernel should be capable of supporting classloader isolation where required.


The Agnostic POC is currently implemented with a MBean adapter and Jetty and Tomcat Launchers, but it would be a small amount of work to launch it from  EAR's or as JBoss MBeans or Gernimo GBeans.

The components are specified with a small XML file that defines a class that activates a component, a classpath in the form of URL's, and a set of component dependencies specified by name.

 

 

The Kernel is started by a LifeCycle Loader that is listening to the lifecycle events in Tomcat or Jetty. It starts the Kernel in its own classloader, that then registers a JMX MBean with the JVM Management Server. This is to enable any class in any classloader to retrieve the kernel without having to use a static factory. Once registered, a ComponentManager and ServiceManager is started. These initialize a Component Registry and a Service Registry respectively. The Component Manager will manage components, and analyze the dependency graphs of components to ensure that all dependencies are satisfied and the start order is correct for the graph. Unlike OSGi, whole components are considered to be the unit of dependency avoiding the need to re-package jars. This does however require that a writer of a component declare the components that they depend on to ensure that those components are started by the Component Manager before the component in question.

Once the Service Manager and Component Manager are started, the Kernel Bootstrap Component is started. This constructs itself using a ComponentActivator that performs the startup and registers the services that the component exports with the ServiceManager. The class that acts as the ComponentActivator for each component is specified by name in the ComponentSpecification. The ComponentSpecification may be presented to the kernel as a special class implementing the ComponentSpecification interface, or it may be a URL to a component XML file; if it is the later, a standard URLComponentSpecification is used to create an instance of the ComponentSpecifcation.

The Kernel Bootstrap component will start a number of things including a SharedClassloaderContainer, which registers a MBean with the JMX MBean server enabling lookup of a suitable configured parent classloader for other application server containers. One such container, is the webapp container.

Once the Kernel bootstrap is started, the ComponentManager loads a set of components. In loading that set of components, it analyzes the dependency DAG, and ensures that all components are resolvable from the list of loaded (but not necessarily started) components. If the DAG analyzer finds that the dependency tree is cyclic, it rejects the component set giving details of the cyclic dependencies. Similarly if the dependency DAG has unsatisfied dependencies it will reject the set of components detailing the missing dependencies. In this analysis, it looks at both the started components and any loaded component specifications. It does not go hunting on the disk to find unsatisfied dependencies.

The dependency DAG analyzer then determines a start order placing root dependencies first in the start order. It then starts each component in turn, resolving its dependencies to ensure that all are started. Having done that the Jetty or Tomcat server lifecycle phase is complete, and the application container may move onto other tasks such as opening connection listeners.

At some point later in the lifecycle the application server will load applications. The load process for the webapp container is modified to have a special classloader implementation added. This looks up the MBean containing the Shared classloader, that was injected in the kernel  bootstrap. This classloader is used to form the parent classloader of the webapp. Finally the webapp looks up the Kernel MBean, and retrieves a reference to the kernel. This is performed via a KernelManager class so the web application doesn't see any of the complexity.


Component Packaging

Components in Sakai2.x were deployed into a components directory in the tomcat tree. In K2 components are packaged as jar's containing a SAKAI-INF/component.xml file that specifies the name of the component, its classpath dependencies; both shared and local to the component. Also specified is the Component Dependencies, and the packages that are exported from the component. The location of the component jars is speficied in the kernel.properties file used to configure the kernel starup.

Component Specification

In the POC I have purposely not bound to any framework technology to make it possible to embed the kernel inside any technology. Consequently the component specification, although it borrows from Gernimo and JBoss is proprietary to Sakai, however its extremely simple. Each component is specific in a ComponentSpecification class either in code, or loaded from an XML file via the ComponentSpefication implementation URLComponentSpecificationImpl. The ComponentSpecificatio specifies the name of the component, which must be unique, a ComponentActivator that is used to start and stop the component, a classpath, and a list of dependent components, specified by name.

Here is a simple example:


<?xml version="1.0" encoding="UTF-8"?>
<component xmlns="http://ns.sakaiproject.org/2008/components">
    <groupId>org.sakaiproject.kernel2.agnostic
    </groupId>
    <artifactId>simplecomponent</artifactId>
    <version>0.1-SNAPSHOT</version>
  <documentation>A simple component with some documentation</documentation>
  <activator>org.sakaiproject.simple</activator>
 </component>
 


Note that there is no classpath, or dependency list specified here. A more complex specification might be:

 

 

<?xml version="1.0" encoding="UTF-8"?>
<component xmlns="http://ns.sakaiproject.org/2008/components">
    <groupId>org.sakaiproject.kernel2.agnostic
    </groupId>
    <artifactId>complextcomponent</artifactId>
    <version>0.1-SNAPSHOT</version>
    <documentation>A more complex component with some documentation
    </documentation>
    <activator>org.sakaiproject.simple</activator>
    <dependencies>
        <dependency>
            <groupId>org.sakaiproject.kernel2.agnostic
            </groupId>
            <artifactId>common</artifactId>
            <version>0.1-SNAPSHOT</version>
            <scope>SHARE</scope>
        </dependency>
    </dependencies>
    <componentDependencies>
        <dependency>
            <groupId>org.sakaiproject.kernel2.agnostic
            </groupId>
            <artifactId>component1</artifactId>
            <version>0.1-SNAPSHOT</version>
            <unmanaged>true</unmanaged>
        </dependency>
        <dependency>
            <groupId>org.sakaiproject.kernel2.agnostic
            </groupId>
            <artifactId>component2</artifactId>
            <version>0.1-SNAPSHOT</version>
            <unmanaged>true</unmanaged>
        </dependency>
        <dependency>
            <groupId>org.sakaiproject.kernel2.agnostic
            </groupId>
            <artifactId>component3</artifactId>
            <version>0.1-SNAPSHOT</version>
            <unmanaged>true</unmanaged>
        </dependency>
    </componentDependencies>
</component>

 

Where this component is names 'complexcomponent' it has a an activator and a single classpath dependency. It also depends on  the component 'component3'.

 

Classloaders

The concepts within the agnostic framework are transportable to other frameworks, but for the moment I have made the assumption that there is no classloader seperation support. This is the case for most webapp containers on their own, but is not the case for a OSGi framework. The following section describes how classloaders are structured in the absence of classloader isolation in the framework.


When the component starts up, if there are classpath dependencies, the component manager will create a classloader, and add the dependencies to that classloader. It will use a DependencyResolverService to convert the ClasspathDependency into a URL which it can add to URLClassloaders. We currently have a Maven2DependencyResolverImpl that knows how to convert the dependency into t a location in maven 2. Its hardly surprising that the dependency part of the component spec is lifted straight from Maven 2 poms. There are two scopes currently available for classpath dependencies, LOCAL and SHARE; LOCAL are added to the component classloader.

 

There is also a shared classloader, this is a custom clasloader that extends the component classloader but it can have dependencies added after it has started. As components start, if they have any SHARE classpath dependencies, these are added to the shared classloader. This shared classloader is availalbe to all components via the SharedClassloaderContainer MBean.


If you look carefully at the XSD below you will notice that the components specification has the concept of export. A component may specify that packages contained within the component may be exported. Examples of exported packages include the component service API's. The component manager on creating the component from the specification will register the exported packages with the PackageRegistryService which builds a package tree, associating packages with component classloaders that contain the packages. Both the Shared and Component Classloaders mentioned above are configured to resolve package names to classloaders using the PackageRegistryService. If a classloader is found, then that classloader will be used to load the class. This happens before the component classloader, shared classloader or an parent classloaders are consulted.

 

By checking for package exports first, it is not possible to override exported packages by mistakenly deploying jars to shared or locally.

 

The full XSD for the component specification is:


 

<?xml version="1.0" encoding="utf-8"?>
<xs:schema xmlns:tns="http://ns.sakaiproject.org/2008/components"
    elementFormDefault="qualified" targetNamespace="http://ns.sakaiproject.org/2008/components"
    xmlns:xs="http://www.w3.org/2001/XMLSchema">
    <xs:complexType name="Component">
        <xs:annotation>
            <xs:documentation>The definition of a component</xs:documentation>
        </xs:annotation>
        <xs:all>
            <xs:element minOccurs="1" name="groupId" type="xs:string" />
            <xs:element minOccurs="1" name="artifactId" type="xs:string" />
            <xs:element minOccurs="1" name="version" type="xs:string" />
            <xs:element minOccurs="0" name="classifier" type="xs:string" />
            <xs:element minOccurs="1" name="documentation" type="xs:string" />
            <xs:element minOccurs="1" name="activator" type="xs:string" />
            <xs:element minOccurs="0" name="componentDependencies"
                type="tns:ArrayOfDependencies" />
            <xs:element minOccurs="0" name="dependencies" type="tns:ArrayOfDependencies" />
            <xs:element minOccurs="0" name="exports" type="tns:ArrayOfExports" />
            <xs:element minOccurs="0" name="unmanaged" type="xs:boolean" default="false" />
        </xs:all>
    </xs:complexType>
    <xs:element name="component" type="tns:Component" />
    <xs:complexType name="ArrayOfDependencies">
        <xs:annotation>
            <xs:documentation>A list of classpath dependencies</xs:documentation>
        </xs:annotation>
        <xs:sequence>
            <xs:element minOccurs="0" maxOccurs="unbounded" name="dependency"
                type="tns:Dependency" />
        </xs:sequence>
    </xs:complexType>
    <xs:complexType name="Dependency">
        <xs:annotation>
            <xs:documentation>Defines a dependency of this component</xs:documentation>
        </xs:annotation>
        <xs:all>
            <xs:element minOccurs="1" name="groupId" type="xs:string" />
            <xs:element minOccurs="1" name="artifactId" type="xs:string" />
            <xs:element minOccurs="1" name="version" type="xs:string" />
            <xs:element minOccurs="0" name="classifier" type="xs:string" />
            <xs:element minOccurs="0" name="scope" type="tns:scope" />
          <xs:element minOccurs="0" name="type" type="xs:string"  default="jar"/>
          <xs:element minOccurs="0" name="unmanaged" type="xs:boolean" default="false" />
        </xs:all>
    </xs:complexType>
    <xs:complexType name="ArrayOfExports">
        <xs:annotation>
            <xs:documentation>A set of exports from this component</xs:documentation>
        </xs:annotation>
        <xs:sequence>
            <xs:element minOccurs="0" maxOccurs="unbounded" name="export"
                type="tns:PackageExport" />
        </xs:sequence>
    </xs:complexType>
    <xs:complexType name="PackageExport">
        <xs:annotation>
            <xs:documentation>Defines packages that should be exported from this component</xs:documentation>
        </xs:annotation>
        <xs:all>
            <xs:element minOccurs="1" name="name" type="xs:string" />
        </xs:all>
    </xs:complexType>
    <xs:simpleType name="scope">
        <xs:annotation>
            <xs:documentation>The Scope of the dependency determining which classloader its added to</xs:documentation>
        </xs:annotation>
        <xs:restriction base="xs:string">
            <!--
                Local to the component Classloader
            -->
            <xs:enumeration value="LOCAL" />
            <!--
                Add to the shared Classloader
            -->
            <xs:enumeration value="SHARE" />
        </xs:restriction>
    </xs:simpleType>
</xs:schema>

 


As you can see, this is relatively simple. Aiming to make it easy for developers to create components. We have not made any assumptions as to what IoC is used how components are constructed to allow developers just to focus on creating components the best way they can.


Service Manager


IMHO one of the failings of the Sakai 2 architecture was not to provide true service registration and isolation. Although Spring is capable of doing this we used a single Spring Application Context for both Compoment Manager and Service registration, in fact, every single bean is availalbe as a bean to all other  components and all webapps as everything is exported. Although we might have liked to have more control over which services were really services Spring prevented us from doing this, at least while we used a single application context for all the component beans. We had to use a single application context for all the beans in all the component becuase we had no formal way of determining or communicating dependencies between components, so we had to use Spring to work out the startup order.

 

In Kernel 2 we have a opportunity to change all that. The Component Manager above, has separated the startup and dependencies of components from the export of service implementations. The Service Manager provides a simple mechanism for allowing components to expose services to one another. Using this approach, only the true services will be exposed. The Search component in Sakai 2x would only expose the SearchService and would not expose the 150 other beans that it needs to construct to be able to function. The User Component in Sakai 2x would only have exposed the UserDirectoryService and not the multitude of other beans.


Service registration (and deregistration) now happens as a result of the ComponentActivator that retrieves the ServiceManager from the kernel, and registers the services that the component wants to export with the ServiceManager. The Component may also use the ServiceManager to resolve Service dependencies.



Current Status


The code kernel is largely complete, with the Component Manager working resolving a DAG of dependencies against component specifications. The kernel bootstrap component works as does service registration and retrieval. All of this is verified by individual unit tests and operation inside Jetty.
